# Migration Decision Log

The auditable record of every migration decision: what was decided, when, by whom, with what evidence. This is the document compliance auditors will ask for 18 months from now when they want to know why SP X was retired and who approved it.

## Rules

1. **Append-only.** Never delete or rewrite a row. If a decision changes, add a new row that supersedes the old one and reference it.
2. **One row per decision**, not per discussion. The discussion is in Slack/PR/email; this log records the outcome.
3. **Every Phase 0 decision MUST appear here** with status `APPROVED` and a date before Phase 1 work begins.
4. **Every SP retirement, every schema deviation from `draft/`, every cutover-eve flip** appears here as it happens.
5. **Evidence is a link.** A PR number, a commit hash, a meeting minutes link, an email, or a screenshot URL. Not "we agreed in the meeting."
6. **The owner is one human**, not a team. Teams cannot be paged.

## How agents use this log

- The `safety-reviewer` subagent checks that any PR retiring an SP includes a row here for that SP.
- The `sp-impact-analyzer` cross-references this log when reporting an SP's history.
- A future `/log-decision` slash command will add rows interactively (not yet built).

## Format conventions

- Dates in ISO 8601: `2026-04-30`.
- Owner = one named person. "Backend team" is not an owner.
- Status values: `pending`, `IN PROGRESS`, `APPROVED`, `REJECTED`, `SUPERSEDED-BY-D##`.
- Evidence: URL or commit hash. If evidence is internal-only, link to the internal system; never paste sensitive content here.
